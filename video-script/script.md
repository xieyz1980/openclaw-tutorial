## 第一章：OpenClaw 是什么？（2分钟）

**【画面：架构概览图】**

**旁白：**

首先，什么是 OpenClaw？

简单来说，OpenClaw 是一个开源的 AI Agent 网关平台。它的核心理念可以用四个关键词概括：

第一，**多平台统一**。无论你使用 WhatsApp、Telegram、Discord、Slack、还是飞书，OpenClaw 都能通过一个统一的接口来管理。

第二，**多模型支持**。你可以自由切换 Claude、GPT、Gemini 等各种 AI 模型，甚至可以为不同任务配置不同的模型。

第三，**可扩展架构**。通过 Skill 系统，你可以轻松扩展 OpenClaw 的功能，就像安装 App 一样简单。

第四，**多代理隔离**。OpenClaw 支持多个完全独立的代理同时运行，每个代理有自己的工作空间、记忆和配置。

**【画面：分层架构动画】**

从架构上看，OpenClaw 分为五个层次：最上层是应用层，包括 WebChat、macOS App、CLI 等客户端；中间是 Gateway 网关层，负责消息路由和协议转换；然后是通道层，连接各种通讯平台；再往下是 Agent 代理层，负责 AI 交互；最底层是 LLM 模型层。

这种分层设计让 OpenClaw 既灵活又强大。

## 第二章：Gateway 网关核心（2分钟）

**【画面：Gateway 架构图 + WebSocket 连接动画】**

**旁白：**

Gateway 是 OpenClaw 的心脏。所有客户端都通过 WebSocket 连接到 Gateway，这保证了实时性和双向通信。

让我们看看一个消息是如何流转的：

首先，用户在 WhatsApp 上发送一条消息。Gateway 收到后，会根据配置的绑定规则，确定这条消息应该路由给哪个 Agent。这个匹配过程非常智能，可以基于频道、账号、用户ID、甚至是群组 ID 来决定。

然后，Gateway 会定位对应的 Session 会话。如果是私聊，会映射到 Agent 的主会话；如果是群组，则使用独立的会话。

接下来，Agent 开始运行。它组装系统提示词，调用 AI 模型，执行工具，生成回复。整个过程是流式的，你可以实时看到 AI 在思考。

最后，回复通过 Gateway 发送回用户。

**【画面：多代理架构图】**

说到多代理，这是 OpenClaw 的一个 killer feature。

假设你有一个家庭代理、一个工作代理、一个编程代理。你可以让 WhatsApp 消息都发给家庭代理，Telegram 发给工作代理，Discord 的某个频道发给编程代理。

每个代理完全隔离：独立的工作目录、独立的会话存储、独立的认证、独立的记忆。这就像在电脑上运行多个独立的虚拟机，互不影响。

## 第三章：工作空间揭秘（2分钟）

**【画面：文件目录树状图】**

**旁白：**

现在让我们打开 OpenClaw 的工作空间，看看里面到底有哪些文件。

首先是 `openclaw.json`，这是主配置文件，采用 JSON5 格式。里面定义了代理列表、通道配置、路由绑定等核心设置。

然后是工作空间目录，这里面有一系列关键的 Markdown 文件：

`AGENTS.md` 是代理的工作规范。每次启动时，OpenClaw 会强制读取这个文件，里面定义了应该首先读取哪些文件、如何管理记忆、安全准则等等。这就像是员工手册，规定了基本工作流程。

`SOUL.md` 是安全规则。这里定义了绝对不可逾越的安全底线：如何防御提示词注入攻击、如何识别恶意技能、哪些操作需要用户确认、哪些目录绝对不能访问。记住，安全永远是第一位的。

`IDENTITY.md` 是角色设定文件。你的代理叫什么名字？是什么性格？有什么职责？这里定义了代理的人格特征。比如我的 IDENTITY 就写着：我是小煤球，一个靠谱、高效、有点小机灵的 AI 助手。

`USER.md` 存储关于用户的信息。用户叫什么名字？应该怎么称呼？在什么时区？有什么特殊偏好？这些信息让代理能够提供个性化的服务。

`MEMORY.md` 是长期记忆。与短期记忆不同，这里的内容会跨会话保留。重要的日期、用户的偏好、正在进行的项目，都应该记录在这里。

`HEARTBEAT.md` 定义定时任务。比如每天下午 5 点自动汇报股市行情，或者每小时检查一次邮件。

`TOOLS.md` 记录环境特定的工具配置。SSH 主机、API 密钥、浏览器配置等，都写在这里作为备忘。

## 第四章：记忆系统深度解析（2分钟）

**【画面：记忆系统架构图】**

**旁白：**

记忆系统是 AI Agent 的灵魂。OpenClaw 采用了双层记忆架构：短期记忆和长期记忆。

短期记忆存在于会话中，包括对话历史、工具调用结果、当前上下文窗口。这些信息随着对话进行不断累积，但会随着会话结束而被清理。

长期记忆则存储在文件中，主要是 `MEMORY.md` 和 `memory/*.md` 每日记录。这些内容会跨会话保留，是代理持续学习的基础。

**【画面：memory_search 流程动画】**

OpenClaw 提供了两个核心工具来操作记忆：

`memory_search` 是语义搜索工具。它会在所有记忆文件中进行语义匹配，找到与查询最相关的片段。你不需要精确的关键词，只需要描述你想找什么。

`memory_get` 用于获取特定文件的特定片段。当 search 找到相关文件后，可以用这个工具读取完整内容。

**【画面：代码示例】**

比如，当用户问"我昨天提到的项目 deadline 是什么时候？"，代理会这样操作：

首先调用 `memory_search`，查询"项目 deadline"。

系统返回匹配的文件路径和相关度分数。

然后调用 `memory_get`，读取那个文件的对应行数。

最后整合信息，给出准确回复。

这种设计既保证了上下文的丰富性，又避免了 token 的浪费。

## 第五章：Skill 系统实战（2分钟）

**【画面：Skill 加载流程动画】**

**旁白：**

如果说记忆是灵魂，那么 Skill 就是 AI Agent 的双手。Skill 让 OpenClaw 能够做更多事情。

**【画面：三个 Skill 目录层级】**

OpenClaw 从三个地方加载 Skill：

首先是 `workspace/skills/`，这是用户工作区的技能，优先级最高。如果你在这里放了一个与内置技能同名的 Skill，它会覆盖内置版本。

然后是 `~/.openclaw/skills/`，这是全局技能目录，对所有代理可见。

最后是内置的 bundled skills，随 OpenClaw 一起安装。

**【画面：SKILL.md 文件示例】**

每个 Skill 都是一个包含 `SKILL.md` 的文件夹。这个文件有固定的结构：

开头是 YAML frontmatter，包含 name、description 和 metadata。metadata 里可以定义门控条件，比如需要哪些环境变量、哪些二进制文件、哪些配置项。

然后是使用说明，告诉代理这个 Skill 是做什么的、怎么用、参数是什么意思。

**【画面：门控表格】**

门控机制是 Skill 系统的智能之处。比如一个视频生成 Skill 需要 `VIDEO_API_KEY` 环境变量，如果用户没有设置，这个 Skill 就不会被加载，代理也就看不到它。

这种设计保证了系统的稳定性和安全性。

## 第六章：实战案例（2分钟）

**【画面：实战案例流程图】**

**旁白：**

理论讲完了，让我们来看一个完整的实战案例：自动按要求生成视频并保存到 GitHub。

**【画面：需求分解动画】**

这个任务需要三步：

第一步，根据用户要求生成视频脚本；
第二步，调用视频生成 API 创建视频；
第三步，将视频推送到 GitHub 仓库。

**【画面：创建 Skill 代码】**

首先，我们创建一个 video-gen Skill。在 `workspace/skills/video-gen/SKILL.md` 中，我们定义：

- 名称：video-gen
- 描述：Generate video from text script
- 门控：需要 VIDEO_API_KEY 环境变量
- 使用方法：调用 curl 发送 POST 请求到视频 API

**【画面：自动化脚本】**

然后，我们编写一个自动化脚本 `auto-video-github.sh`：

脚本首先接收用户输入的主题，然后生成视频脚本，调用 API，等待视频生成完成，下载视频文件，最后使用 gh CLI 推送到指定的 GitHub 仓库。

**【画面：用户交互示例】**

使用时，用户只需要说："帮我制作一个关于 OpenClaw 入门的视频，保存到 github.com/xieyz1980/videos"

代理会执行整个流程，并在完成后返回 GitHub 链接。

这就是 OpenClaw 的强大之处：通过组合不同的 Skill，你可以构建复杂的自动化工作流。

好了，让我们总结一下今天学到的内容。

OpenClaw 的核心架构包括：Gateway 网关负责统一接入和路由，工作空间通过一系列 Markdown 文件定义代理的行为和记忆，Skill 系统提供可扩展的能力，双层记忆系统平衡了上下文丰富度和 token 效率。

**【画面：最佳实践列表】**

几点最佳实践：

保持注入文件精简，因为它们每次都会消耗 token。

定期整理记忆，将重要信息从每日记录迁移到长期记忆。

善用 Skill 扩展功能，把复杂操作封装成可复用的 Skill。

始终遵循 SOUL.md 的安全规则，安全第一。

最后，将 workspace 纳入版本控制，这样你可以追踪配置的变化。
